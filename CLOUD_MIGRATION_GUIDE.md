# Cloud Migration Guide — ListingJet

## Status: NOT YET DEPLOYED — Save for when you're ready

## Will frontend/UI changes affect this?

**No.** Frontend and cloud migration are fully independent:

- **Frontend** → Deployed on Vercel. Changes to React/Next.js pages, components, styles have zero impact on AWS infrastructure.
- **Backend API** → Already containerized via Dockerfile. Any new endpoints, models, or services get picked up automatically on the next Docker build + ECS deploy.
- **CDK infra** → Only changes when you modify files in `infra/`. UI work never touches these.

**Keep building your UI.** When you're ready to go live, come back to this guide.

## Remaining Gaps Before Deploy

### Gap 1: Temporal — Consider Temporal Cloud

**Current:** Self-hosted `temporalio/auto-setup` on Fargate (desired_count=1).
**Option:** Move to Temporal Cloud (free tier covers early usage) to reduce operational burden.

**If moving to Temporal Cloud:**
- `infra/stacks/services.py` — Remove the Temporal Fargate service
- `infra/stacks/services.py` — Update `base_env` to use Temporal Cloud endpoint:
  ```python
  "TEMPORAL_HOST": "<your-namespace>.tmprl.cloud:7233",
  "TEMPORAL_NAMESPACE": "<your-namespace>",
  ```
- Add Temporal Cloud API key to secrets
- Sign up at https://cloud.temporal.io — free tier gives 1K actions/month

**If keeping self-hosted:** It's already configured and running. Just verify DB connectivity after RDS deploys.

### Gap 2: Populate AWS Secrets Manager

**Before first deploy, create the secret:**

```bash
aws secretsmanager create-secret \
  --name "listingjet/app" \
  --secret-string '{
    "DATABASE_URL": "will-be-filled-after-rds-deploys",
    "JWT_SECRET": "generate-a-64-char-random-string",
    "STRIPE_SECRET_KEY": "sk_live_...",
    "STRIPE_WEBHOOK_SECRET": "whsec_...",
    "SENTRY_DSN": "",
    "OPENAI_API_KEY": "sk-...",
    "ANTHROPIC_API_KEY": "sk-ant-...",
    "GOOGLE_VISION_API_KEY": "...",
    "KLING_ACCESS_KEY": "...",
    "KLING_SECRET_KEY": "..."
  }'
```

**After RDS deploys**, update DATABASE_URL with the actual RDS endpoint:
```bash
aws secretsmanager update-secret \
  --secret-id "listingjet/app" \
  --secret-string '{
    "DATABASE_URL": "postgresql+asyncpg://listingjet:<password>@<rds-endpoint>:5432/listingjet",
    ...rest of keys...
  }'
```

The RDS password is auto-generated by CDK and stored in a separate secret (`ListingJetDatabase/Postgres/Secret`).

### Gap 3: Vercel API URL

**Current:** `vercel.json` rewrites `/api/*` to the ALB endpoint.
**Action:** After deploying, update the ALB URL in `vercel.json` (or move to `next.config.js` rewrites with env var):

```js
// frontend/next.config.js
module.exports = {
  async rewrites() {
    return [
      {
        source: '/api/:path*',
        destination: `${process.env.API_ORIGIN || 'http://localhost:8000'}/:path*`,
      },
    ];
  },
};
```

Then set `API_ORIGIN` in Vercel dashboard → Settings → Environment Variables.

### Gap 4: Domain + HTTPS (Optional for Beta)

The ALB already has HTTPS support via the `domain_name` CDK context. To enable:

```bash
cdk deploy ListingJetServices -c domain_name=api.listingjet.com
```

This creates an ACM certificate, adds an HTTPS listener, and redirects HTTP→HTTPS. Requires a domain with DNS validation (Route53 or manual).

For beta, HTTP on the raw ALB URL works fine.

## Deploy Sequence (when ready)

```bash
# 1. Install CDK
npm install -g aws-cdk

# 2. Configure AWS credentials
aws configure  # or use SSO

# 3. Bootstrap CDK (first time only)
cd infra
pip install -r requirements.txt
cdk bootstrap aws://<ACCOUNT_ID>/us-east-1

# 4. Deploy stacks in order
cdk deploy ListingJetNetwork
cdk deploy ListingJetDatabase
# → Note the RDS endpoint from outputs
# → Create/update the Secrets Manager secret with real DATABASE_URL

cdk deploy ListingJetServices
cdk deploy ListingJetCDN
cdk deploy ListingJetMonitoring
cdk deploy ListingJetCI

# 5. Push first Docker image
# (or just push to main — GitHub Actions handles it via deploy.yml)

# 6. Run initial migration
aws ecs run-task \
  --cluster listingjet \
  --task-definition listingjet-api \
  --overrides '{"containerOverrides":[{"name":"api","command":["python","-m","alembic","upgrade","head"]}]}'

# 7. Update Vercel env var with ALB/domain URL
# Vercel Dashboard → Settings → Environment Variables → API_ORIGIN
```

## Things You Do NOT Need to Worry About

- **ClamAV** — Use `USE_MOCK_PROVIDERS=true` or skip virus scanning for beta. Add Lambda-based scanning later.
- **Jaeger** — You already have Sentry + OpenTelemetry configured. CloudWatch Container Insights covers the rest.
- **Docker on your PC** — Once deployed, `docker compose up postgres redis temporal -d` is still useful for local dev, but production runs independently.
